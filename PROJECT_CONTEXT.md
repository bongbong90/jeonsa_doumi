## 1. 프로젝트 개요

전사도우미는 Windows 데스크톱 환경에서 MP3 파일을 텍스트로 전사하기 위한 업무용 프로그램이다.

주요 목적은 다음과 같다.

- MP3 강의/녹음 파일을 텍스트 파일로 전사
- SRT 자막 파일 생성
- 여러 MP3 파일을 폴더 단위로 순차 전사
- 전사 진행률과 현재 처리 파일 표시
- 전사 완료 알림 제공
- 시스템 트레이 최소화 및 복원 지원
- 전체 전사 완료 후 컴퓨터 종료 옵션 제공

이 프로젝트는 개인 업무 및 학습 자료 정리를 위한 실사용 도구이며, 디자인보다 안정적인 전사 처리와 결과물 생성이 우선이다.

- 전사 진행률 표시
- 현재 처리 중 파일 표시
- 현재 파일 진행률 표시
- 로그창 표시
- 즉시 중지 요청
- 트레이 최소화/복원 처리

### auto_transcribe.py

실제 전사 엔진 역할을 담당한다.

주요 역할:

- Whisper 모델 로드
- MP3 파일 전사

- srt 결과물 생성
- 세션 상태 저장/복구 관련 처리
- stop.flag 감지 기반 중지 처리

### 전사도우미.spec

PyInstaller 빌드 설정 파일이다.

현재 프로젝트는 onefile 방식이 아니라 onedir 방식을 기준으로 한다.

## 4. GUI와 전사 엔진 역할 분리

이 프로젝트는 GUI와 전사 엔진을 분리하는 구조다.

- GUI는 실행 제어, 중지 요청, 진행률 표시, 파일 이동, 로그 표시를 담당한다.
- 전사 엔진은 `auto_transcribe.py`가 담당한다.
- GUI는 `auto_transcribe.py`를 호출하고, stdout 로그와 진행률 정보를 파싱하여 화면에 반영한다.

따라서 오류 발생 시 먼저 아래를 구분해야 한다.

1. GUI 쪽 문제인지
2. `auto_transcribe.py` 쪽 문제인지
3. exe 환경 문제인지
4. `.py` 직접 실행 환경 문제인지

## 5. 진행률 처리 방식

현재 진행률 처리는 `progress_callback` 방식이 아니다.

과거 `progress_callback` 사용을 검토했으나, 사용자 PC의 openai-whisper 환경에서 지원되지 않아 오류가 발생했다.

따라서 현재 구조는 stdout에 출력되는 진행률 퍼센트를 GUI가 파싱하는 방식이다.

진행률 관련 문제 발생 시 우선 확인할 것:

- stdout 파싱 로직
- 현재 파일 진행률 파싱 기준
- 전체 진행률 계산 기준
- 완료 파일 수 계산 기준
- 스킵 파일 수 계산 기준
- GUI 진행률 분모 기준

## 6. 로그 처리 방식

로그는 GUI 로그창 기준으로 사용한다.

현재 정책:

- 별도 `.log` 파일 자동 저장은 사용하지 않음
- GUI 로그창에서 실행 로그 확인
- 필요 시 로그창 보기/숨기기
- 콘솔 한글 깨짐이 있을 수 있으므로 GUI 로그를 우선 신뢰

한글 깨짐 문제 대응:

- GUI 쪽에서 `utf-8` / `cp949` / `euc-kr` 순으로 디코딩 시도
- 콘솔 출력보다 GUI 로그를 기준 로그로 사용

## 7. 즉시 중지 방식

즉시 중지는 `stop.flag` 파일 기반으로 동작한다.

현재 구조:

- GUI에서 즉시 중지 버튼 클릭
- `stop.flag` 생성
- 전사 프로세스 terminate 시도
- 종료되지 않으면 kill까지 수행

현재 확인된 문제:

- 사용자가 즉시 중지를 눌렀는데 terminate 후 종료되지 않아 kill()이 실행됨
- 이 경우 `normal=False`로 처리됨
- 사용자 중지임에도 다음 실행 시 비정상 종료 흔적으로 감지될 수 있음

향후 수정 필요:

- 사용자 중지와 실제 비정상 종료를 구분
- `stop.flag` 감지 주기 개선
- 사용자 중지 상태를 세션 상태에 명확히 기록
- 중지 중이던 파일은 완료 처리하지 않도록 보장

## 8. 재실행 복구 방식

전사 도중 중지되거나 비정상 종료될 수 있으므로 재실행 복구 구조가 존재한다.

기본 방향:

- session state와 현재 진행 파일 정보를 기준으로 미완료 파일을 재처리
- 이미 완료된 파일은 스킵
- 미완료 파일은 다시 전사 대상에 포함

주의:

- 세션 상태 저장 실패가 발생하면 재실행 복구와 진행률 계산이 꼬일 수 있음
- 완료 파일과 미완료 파일 판정 기준을 명확히 해야 함
- 불완전한 txt/srt 결과물을 완료로 오인하면 안 됨

## 9. exe 배포 구조

현재 배포 구조는 onedir 방식이다.

최종 실행 구조:

`dist\전사도우미\`

- `전사도우미.exe`
- `auto_transcribe.py`
- `_internal\`

주의사항:

- `auto_transcribe.py`는 `전사도우미.exe`와 같은 폴더에 있어야 함
- `_internal` 폴더는 삭제하거나 이동하면 안 됨
- onefile 방식은 현재 구조와 맞지 않음
- frozen 환경에서는 `sys.executable` 기준으로 경로 계산해야 함
- 일반 Python 실행 환경에서는 `gui_main.py` 위치 기준으로 경로 계산해야 함

## 10. exe 실행 관련 과거 이슈

과거 발생한 주요 이슈:

1. onefile 빌드 경로 오류
   - `auto_transcribe.py`를 임시 `_MEI` 경로에서 찾는 문제 발생
   - onedir 방식으로 전환
2. onedir 환경에서도 `auto_transcribe.py` 경로 오류
   - frozen 환경에서 `sys.executable` 기준 경로 계산으로 수정
3. exe가 자기 자신을 다시 실행하는 문제
   - exe 환경에서 `sys.executable`이 `전사도우미.exe` 자체를 가리킴
   - QProcess가 `auto_transcribe.py`가 아니라 exe를 다시 실행하는 문제 발생
   - exe 환경에서는 python 명령으로 `auto_transcribe.py` 실행하도록 분기

향후 exe 관련 문제 발생 시 위 내용을 우선 확인한다.

## 11. 현재 확인된 주요 오류

### 11-1. 세션 상태 저장 실패

전사 중 아래 경고가 반복 발생했다.

`[WARN] 세션 상태 저장 실패: [WinError 5] 액세스가 거부되었습니다: transcribe_session_state.json.tmp -> transcribe_session_state.json`

예상 문제:

- GUI와 worker가 같은 세션 파일을 동시에 접근할 가능성
- Windows 파일 잠금 문제
- tmp 파일 replace 과정에서 `PermissionError` 발생
- 세션 파일 핸들이 완전히 닫히지 않았을 가능성
- `os.replace` 또는 rename 처리 시 파일 잠금 충돌 가능성

### 11-2. 이전 작업 비정상 종료 흔적 감지

실행 시작 시 아래 경고가 발생했다.

`[WARN] 이전 작업 비정상 종료 흔적 감지`

예상 문제:

- 이전 실행의 세션 상태가 정상 종료로 정리되지 않음
- 사용자 중지와 실제 비정상 종료가 구분되지 않음
- 세션 상태 저장 실패로 인해 정상 종료 상태가 기록되지 않음

### 11-3. 즉시 중지 시 강제 kill 발생

즉시 중지 후 아래 로그가 확인되었다.

- `[GUI] stop.flag 생성 완료 - 즉시 중지 요청`
- `[GUI] terminate 후 종료되지 않아 kill() 실행`
- `[GUI] 프로세스 종료: exit_code=62097, normal=False`

예상 문제:

- `stop.flag` 생성은 되었으나 `auto_transcribe.py`가 즉시 반응하지 못함
- worker가 `stop.flag`를 자주 확인하지 않음
- 사용자 중지임에도 `normal=False`로 처리됨
- 다음 실행 시 비정상 종료로 감지될 수 있음

## 12. 현재 작업 우선순위

현재 우선순위는 아래와 같다.

1. 프로젝트 폴더 구조 정리
2. `PROJECT_CONTEXT.md` 생성
3. 세션 상태 저장 실패 오류 수정
4. 즉시 중지 시 사용자 중지와 비정상 종료 구분
5. 진행률 기준 정리

현재 디자인 작업은 아직 진행하지 않는다. 먼저 오류 안정화 작업을 완료한 뒤 디자인 작업을 진행한다.
