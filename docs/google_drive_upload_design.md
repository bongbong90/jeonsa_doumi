# Google Drive 자동 업로드 기능 설계 문서

## 1. 개요
전사 완료 후 생성된 4종 파일(MP3, TXT, JSON, SRT)을 사용자의 Google Drive에 자동 업로드하는 기능을 구현한다. 로컬 전사 처리 이후, 기존에 정립된 Google Drive 폴더 구조에 맞추어 자동으로 분류하고 저장하는 것을 목표로 한다.

## 2. 기존 Google Drive 구조
사용자가 관리하는 실제 Drive 구조를 최우선으로 사용하여 업로드 경로를 매핑한다. 각 과목 폴더 안에는 강의별 폴더가 아닌 주차별 폴더가 존재한다.

```text
내 드라이브/
└─ 2026 제37회 공인중개사 자격시험/
   ├─ 기출문제/
   ├─ 복습자료/
   └─ 전사자료/
      ├─ 개념완성/
      │  ├─ [1차] 민법/
      │  │  ├─ 개념완성_민법_1주차/
      │  │  ├─ 개념완성_민법_2주차/
      │  │  ├─ 개념완성_민법_3주차/
      │  │  └─ 개념완성_민법_4주차/
      │  ├─ [1차] 부동산학개론/
      │  ├─ [2차] 공인중개사법/
      │  ├─ [2차] 부동산공법/
      │  ├─ [2차] 부동산공시법/
      │  └─ [2차] 부동산세법/
      ├─ 기본이론/
      └─ 기초이론/
```

## 3. 업로드 경로 설계
새로운 루트 폴더를 생성하지 않고, 기존 `내 드라이브/2026 제37회 공인중개사 자격시험/전사자료` 폴더를 기본 루트로 활용한다. 강의별 하위 폴더는 별도로 만들지 않는다.

**업로드 경로 규칙:**
`내 드라이브/2026 제37회 공인중개사 자격시험/전사자료/[과정명]/[과목 폴더]/[주차 폴더]/`

**예시:** `개념완성_민법_1주차_1강.mp3` 파일 전사 시
```text
내 드라이브/
└─ 2026 제37회 공인중개사 자격시험/
   └─ 전사자료/
      └─ 개념완성/
         └─ [1차] 민법/
            └─ 개념완성_민법_1주차/
               ├─ 개념완성_민법_1주차_1강.mp3
               ├─ 개념완성_민법_1주차_1강.txt
               ├─ 개념완성_민법_1주차_1강.json
               ├─ 개념완성_민법_1주차_1강.srt
               ├─ 개념완성_민법_1주차_2강.mp3
               ├─ 개념완성_민법_1주차_2강.txt
               ├─ 개념완성_민법_1주차_2강.json
               └─ 개념완성_민법_1주차_2강.srt
```

## 4. 업로드 대상 파일
해당 주차 폴더 안에 다음 4종의 파일을 직접 일괄 업로드한다.
1. **원본 MP3**: 대용량 파일이므로 Google Drive API의 Resumable Upload(이어올리기) 방식을 기본으로 사용한다.
2. **전사 TXT**
3. **전사 JSON**
4. **전사 SRT**

## 5. 자동 분류 기준

### 5.1. 과정명 자동 분류
파일명 또는 로컬 폴더 경로에서 다음 키워드를 감지하여 매핑한다.
- `개념완성` → 전사자료/개념완성
- `기본이론` → 전사자료/기본이론
- `기초이론` → 전사자료/기초이론

**과정명 감지 실패 시 정책:**
- 기본값으로 `전사자료/분류대기` 폴더를 생성하여 사용한다.
- 또는 GUI 설정에서 사용자가 '기본 업로드 과정명'을 직접 지정할 수 있도록 지원한다.

### 5.2. 과목 자동 분류
파일명 또는 로컬 경로 키워드에 따라 과목 폴더를 매핑한다.
- `민법`, `김덕수` → `[1차] 민법`
- `부동산학개론`, `학개론`, `개론`, `이종호` → `[1차] 부동산학개론`
- `공인중개사법`, `중개사법`, `중개실무`, `정지웅` → `[2차] 공인중개사법`
- `부동산공법`, `공법`, `김희상` → `[2차] 부동산공법`
- `부동산공시법`, `공시법`, `등기법`, `지적법`, `박윤모` → `[2차] 부동산공시법` (⚠️ `공법` 키워드와 겹치지 않도록 먼저 평가)
- `부동산세법`, `세법`, `정석진` → `[2차] 부동산세법`

**과목 감지 실패 시 정책:**
- `과목불명` 또는 `분류대기` 폴더를 생성하여 해당 위치로 업로드한다.

### 5.3. 주차 폴더 감지 정책
파일명 또는 로컬 경로에서 주차를 의미하는 패턴을 감지하여 해당 주차 폴더로 분류한다.
- 예: `개념완성_민법_1주차_1강.mp3` → 주차 폴더: `개념완성_민법_1주차`
- 예: `개념완성_민법_4주차_3강.mp3` → 주차 폴더: `개념완성_민법_4주차`
- 예: `기본이론_공법_7주차_2강.mp3` → 주차 폴더: `기본이론_공법_7주차`
- 예: `기초이론_세법_2주차_1강.mp3` → 주차 폴더: `기초이론_세법_2주차`

**감지 로직 및 실패 시 정책:**
- 파일 stem에서 `_숫자주차_숫자강` 패턴을 정규식 등으로 찾는다.
- `..._1주차_3강` 형태이면 `_3강` 앞부분까지를 잘라 주차 폴더명으로 사용한다.
- 감지 실패 시 파일 stem을 그대로 쓰지 않고 `주차불명` 폴더로 업로드하거나 GUI 설정값을 사용한다.

## Google Drive 업로드 전 파일명 정규화 및 진행률 개선 정책
Google Drive 자동 업로드 전에 다운로드 원본 파일명, 전사 완료 파일명, Drive 업로드 파일명이 서로 달라지는 문제가 확인되었다. 사용자는 현재 Drive 업로드 전에 파일명을 직접 `과정명_과목명_N주차_N강` 형식으로 수정하고 있으며, 전사 완료 파일명에는 페이지 표기가 남아 있고 MP3 원본 파일명은 유지되는 반면 TXT/JSON/SRT만 일부 정제된 이름으로 생성될 수 있다. 또한 CURRENT PROGRESS 바와 작업 트레이/Toast 진행률이 전사 중 충분히 갱신되지 않고, REMAINING TIME이 실제보다 과하게 길게 표시되는 문제가 함께 확인되었다.

### 파일명 용도 구분
파일명은 다음 3가지 개념으로 명확히 구분한다.

1. **다운로드 원본 파일명**
   - 강의 사이트에서 내려받은 긴 원본명이다.
   - 페이지, 날짜, 교재명, 제목, `+` 기호 등이 포함될 수 있다.
   - 원본명은 과정/과목/주차/강 번호 감지와 제목 분석 참고용으로만 사용한다.
   - 세법 예: `12강_[4주차]_26_03_30_[교재]+2주차+복습+과정_+양도소득+과세표준과+세액의+계산+(p.+40)+_+국외자산양도에+대한+양도소득세+(p.+70+~+).mp3`
   - 부동산학개론 예: `13강_[4-1] 시장론 4.mp3`

2. **제목형 정리 파일명**
   - 페이지 표기를 제거한다.
   - `+` 기호를 제거하거나 공백으로 정리한다.
   - 공백과 언더바를 정리한다.
   - 강의 제목을 보존하는 사람이 읽기 쉬운 정리명이다.
   - 단, Google Drive 자동 업로드 기준명으로는 사용하지 않는다.

3. **표준 업로드/전사 파일명**
   - Google Drive 업로드 및 전사 결과 저장의 기준 파일명이다.
   - 형식은 `과정명_과목명_N주차_N강`으로 통일한다.
   - 예: `개념완성_부동산세법_4주차_1강`
   - 예: `개념완성_부동산학개론_4주차_1강`
   - 예: `기본이론_민법_3주차_2강`
   - 예: `기초이론_공인중개사법_2주차_1강`

### 페이지 표기 전체 제거
- 전사 완료 후 파일명에서는 페이지 표기가 모두 제거되어야 한다.
- 파일명 끝뿐 아니라 중간에 있는 페이지 표기도 제거 대상이다.
- 향후 함수는 `remove_page_suffix()` 수준이 아니라 `remove_page_markers()` 또는 동등한 역할로 확장한다.
- `auto_transcribe.py`와 `gui_main.py`에 중복된 파일명 정리 함수가 있다면 같은 규칙으로 맞춘다.
- 제거 대상 예:
  - `(p.40)`
  - `(p. 40)`
  - `(p.+40)`
  - `(p.70~)`
  - `(p.+70+~+)`
  - `(p.123~456)`
  - `(p.+123+~+456)`

### 기호/공백 정리
- `+`는 공백으로 변환한다.
- 연속 공백은 1칸으로 축소한다.
- `_` 주변 공백을 정리한다.
- 연속 `_`를 정리한다.
- 앞뒤 공백과 불필요한 `_`를 제거한다.
- Windows 파일명 금지문자는 안전한 문자로 치환하거나 제거한다.
- 예: `과정_ 양도소득 과세표준과 세액의 계산  _ 국외자산양도` → `과정_양도소득 과세표준과 세액의 계산_국외자산양도`

### 표준 파일명 자동 생성
Google Drive 자동 업로드 전 표준 파일명을 자동 생성한다. 표준 형식은 `과정명_과목명_N주차_N강`이다.

1. **과정명**
   - `개념완성`, `기본이론`, `기초이론`을 감지한다.
   - 감지 실패 시 `분류대기` 또는 GUI 사용자 선택값을 사용한다.

2. **과목명**
   - 대상 과목은 `민법`, `부동산학개론`, `공인중개사법`, `부동산공법`, `부동산공시법`, `부동산세법`이다.
   - 감지 실패 시 `과목불명` 또는 GUI 사용자 선택값을 사용한다.
   - `공시법`은 `공법`보다 먼저 매칭한다.
   - 과목 감지 키워드:
     - `민법`, `김덕수` → `민법`
     - `부동산학개론`, `학개론`, `개론`, `이종호` → `부동산학개론`
     - `공인중개사법`, `중개사법`, `중개실무`, `정지웅` → `공인중개사법`
     - `부동산공시법`, `공시법`, `등기법`, `지적법`, `박윤모` → `부동산공시법`
     - `부동산공법`, `공법`, `김희상` → `부동산공법`
     - `부동산세법`, `세법`, `정석진` → `부동산세법`

3. **주차**
   - 파일명 또는 상위 폴더명에서 감지한다.
   - 예: `[4주차]`, `[4-1]`, `개념완성_민법_4주차`, `4주차`
   - 감지 실패 시 `주차불명` 또는 GUI 사용자 선택값을 사용한다.

4. **강 번호**
   - 원본 파일명 앞의 `12강`, `13강` 같은 전체 강의 번호를 표준 파일명의 강 번호로 그대로 쓰지 않는다.
   - Google Drive 업로드 기준은 해당 주차 안에서의 순번이다.
   - 같은 주차 폴더 안 MP3를 natural sort한 뒤 첫 번째 MP3는 `1강`, 두 번째 MP3는 `2강`, 세 번째 MP3는 `3강`으로 부여한다.
   - 단, 부동산학개론처럼 `[N-M]` 패턴이 명확한 경우에는 해당 패턴을 우선 사용한다.
   - 원본 앞의 `12강`, `13강` 등은 전체 강의 번호로 보고 정렬 참고값으로만 사용할 수 있다.

세법 예:
- 원본: `12강_[4주차]_26_03_30_[교재]+...mp3`
- 표준명: `개념완성_부동산세법_4주차_1강.mp3`

부동산학개론 예:
- 원본: `13강_[4-1] 시장론 4.mp3`
- 표준명: `개념완성_부동산학개론_4주차_1강.mp3`

### 과목별 다운로드 파일명 패턴 차이
파일명 정규화 로직은 특정 다운로드 파일명 패턴 하나에 고정하지 않고, 과목별 패턴 차이를 허용하는 다중 감지 방식으로 설계한다.

- 과목별 다운로드 파일명 패턴은 서로 다를 수 있다.
  - 부동산세법처럼 `12강_[4주차]_...` 형태가 있을 수 있다.
  - 부동산학개론처럼 `13강_[4-1] 시장론 4.mp3` 형태가 있을 수 있다.
  - 다른 과목도 교수/사이트/자료 유형에 따라 파일명 구조가 다를 수 있다.
- 표준 파일명 생성은 원본 파일명 하나만으로 결정하지 않고 다음 정보를 우선순위에 따라 함께 사용한다.
  1. 사용자가 선택한 폴더 경로
  2. 상위 폴더명
  3. 파일명 내 과정/과목/주차 키워드
  4. 파일명 내 명확한 주차-강 패턴
  5. GUI에서 사용자가 지정한 기본 과정/과목/주차 값
  6. 감지 실패 시 `분류대기`, `과목불명`, `주차불명`
- 주차/강 번호 추출은 과목별 원본 파일명 패턴 하나에 의존하지 않는다.
  - 원본 파일명에 `12강`, `13강` 같은 전체 강의 번호가 있더라도 최종 표준 강 번호로 바로 사용하지 않는다.
  - 원본 파일명에 주차 정보가 없거나 패턴이 다르면 상위 폴더명 또는 GUI 사용자 선택값을 우선 사용한다.
  - 최종 강 번호는 같은 주차 폴더 안 MP3 파일을 natural sort한 순서로 `1강`, `2강`, `3강`처럼 부여하는 것을 기본 정책으로 한다.
  - 단, `[N-M]`처럼 주차와 주차 내 강 번호가 명확한 패턴은 우선 사용한다.
- 향후 구현 단계에서 과목별 샘플 파일명 테스트 케이스를 수집한다.
  - 부동산세법 다운로드 파일명 예시
  - 부동산학개론 다운로드 파일명 예시
  - 민법 다운로드 파일명 예시
  - 부동산공법/부동산공시법처럼 명칭이 헷갈릴 수 있는 과목 예시

### 부동산학개론 확정 파일명 패턴
부동산학개론 실제 파일명 예시가 확인되었으므로 다음 샘플과 해석 정책을 확정 기준으로 반영한다.

1. **다운로드 파일명 패턴 예시**
   - 다운로드 시 파일명: `13강_[4-1] 시장론 4.mp3`
   - 전사 완료 시 현재 파일명: `13강_[4-1] 시장론 4.mp3`
   - Google Drive 업로드 시 사용자가 수동으로 바꾸던 표준 파일명: `개념완성_부동산학개론_4주차_1강.mp3`

2. **`[N-M]` 패턴 해석 정책**
   - `[4-1]` → `4주차_1강`
   - `[4-2]` → `4주차_2강`
   - `[5-1]` → `5주차_1강`
   - 부동산학개론에서 `[N-M]` 패턴이 명확하게 존재하면 `N = 주차`, `M = 해당 주차 내 강 번호`로 우선 해석한다.

3. **전체 강의 번호와 주차 내 강 번호 구분**
   - 부동산학개론 예시의 `13강`은 전체 강의 번호로 볼 수 있으므로 표준 파일명의 강 번호로 직접 사용하지 않는다.
   - 원본 앞부분: `13강`
   - 중간 패턴: `[4-1]`
   - 표준 파일명: `개념완성_부동산학개론_4주차_1강.mp3`
   - 따라서 `13강`을 표준 파일명의 `13강`으로 사용하지 않는다.

4. **표준 파일명 생성 예시**
   - 원본: `13강_[4-1] 시장론 4.mp3`
   - 표준명: `개념완성_부동산학개론_4주차_1강.mp3`
   - Drive 업로드 위치: `2026 제37회 공인중개사 자격시험/전사자료/개념완성/[1차] 부동산학개론/개념완성_부동산학개론_4주차/`

5. **강 번호 자동 부여 우선순위**
   1. 파일명에 `[N-M]` 패턴이 명확한 경우 → `N주차_M강`
   2. 파일명에 `N주차_M강` 패턴이 있는 경우 → `N주차_M강`
   3. 상위 폴더에서 주차를 감지하고 MP3 natural sort 순서로 강 번호 부여
   4. GUI 사용자 선택값 사용
   5. 감지 실패 시 `주차불명` 또는 사용자 확인 필요

### 전사 시작 전 MP3 rename 및 결과 파일 stem 일치
- 전사 시작 전에 MP3 원본 파일명을 표준 파일명으로 rename한다.
- rename된 MP3 파일을 기준으로 전사를 수행한다.
- TXT/JSON/SRT는 rename된 MP3와 같은 stem으로 저장한다.
- Google Drive 업로드도 같은 stem의 MP3/TXT/JSON/SRT 4종 파일을 대상으로 한다.
- 목표 결과 예:
  - `개념완성_부동산세법_4주차_1강.mp3`
  - `개념완성_부동산세법_4주차_1강.txt`
  - `개념완성_부동산세법_4주차_1강.json`
  - `개념완성_부동산세법_4주차_1강.srt`
- 목표 결과 예:
  - `개념완성_부동산학개론_4주차_1강.mp3`
  - `개념완성_부동산학개론_4주차_1강.txt`
  - `개념완성_부동산학개론_4주차_1강.json`
  - `개념완성_부동산학개론_4주차_1강.srt`

### rename 미리보기, 확인, 충돌 처리
- 자동 rename은 위험하므로 전사 시작 전에 변경 미리보기를 제공한다.
- 미리보기 예:
  - 변경 전: `12강_[4주차]_26_03_30_[교재]+...mp3`
  - 변경 후: `개념완성_부동산세법_4주차_1강.mp3`
- 미리보기 예:
  - 변경 전: `13강_[4-1] 시장론 4.mp3`
  - 변경 후: `개념완성_부동산학개론_4주차_1강.mp3`
- 변경 대상이 있으면 사용자 확인 후 rename한다.
- 사용자가 취소하면 전사를 시작하지 않는다.
- 표준명으로 rename할 때 같은 이름이 이미 있으면 자동 덮어쓰기를 금지한다.
- 충돌 파일이 있으면 rename을 중단하고 충돌 목록을 표시한다.
- 대안으로 `_dup1`, `_dup2` suffix를 부여할 수 있으나, 1차 구현에서는 자동 suffix보다 충돌 중단을 우선 권장한다.

### 기존 결과 파일 동기화
- 이미 결과 파일이 있는 상태에서 MP3 rename이 필요할 수 있다.
- 같은 stem의 기존 TXT/JSON/SRT가 있으면 MP3 rename과 함께 같이 rename하는 방식을 고려한다.
- 단, 전사 시작 전 신규 파일이면 MP3만 rename해도 결과 파일은 새 stem 기준으로 생성된다.
- 기존 결과물 존재 여부와 rename 대상 판단을 명확히 분리한다.

### Google Drive 업로드와의 관계
- Google Drive 업로드는 표준화된 파일명을 기준으로 수행한다.
- 표준화되지 않은 MP3는 업로드 전에 정규화 대상이다.
- `google_drive_uploader.py`의 과정/과목/주차 감지는 표준 파일명을 기준으로 안정적으로 동작하도록 설계한다.
- 표준 파일명 미적용 상태에서 Drive 업로드를 진행하지 않는 방향으로 설계한다.

### CURRENT PROGRESS 및 트레이 알림 진행률 동기화
- 현재 로컬 Whisper 전사 중 CURRENT PROGRESS 바가 전사 중 채워지지 않는 문제가 있다.
- 사용자가 보기에는 프로그램이 멈춘 것처럼 보일 수 있으므로 Google Drive 자동 업로드 전 현재 파일 진행률 표시를 개선한다.
- 로컬 Whisper는 정확한 실시간 진행률을 제공하기 어렵기 때문에 1차 구현에서는 단계 기반 진행률을 사용한다.
- 권장 단계:
  - 파일 시작: 5%
  - 모델 준비 완료: 15%
  - Whisper 전사 시작: 20%
  - Whisper 전사 완료: 75%
  - corrections 적용 완료: 85%
  - TXT/JSON/SRT 저장 완료: 95%
  - `FILE_DONE` 수신: 100%
- CURRENT PROGRESS를 수정할 때 메인 화면의 현재 파일 진행률 바만 수정하지 않고 작업 트레이 알림/Toast에 표시되는 진행률 바도 같은 기준으로 동기화한다.
- 동기화 대상:
  - 메인 화면 CURRENT PROGRESS 바
  - 메인 화면 현재 파일 퍼센트 텍스트
  - 작업 트레이/Toast 진행률 바
  - 작업 트레이/Toast 진행률 퍼센트 텍스트
  - 현재 파일명 표시
- `auto_transcribe.py`에서 진행 단계 이벤트를 출력하고, `gui_main.py`가 이를 받아 메인 UI와 Toast UI를 같이 갱신하는 방향으로 설계한다.
- 예시 이벤트:
  - `FILE_PROGRESS|filename|5|파일 시작`
  - `FILE_PROGRESS|filename|20|Whisper 전사 시작`
  - `FILE_PROGRESS|filename|75|Whisper 전사 완료`
  - `FILE_PROGRESS|filename|85|corrections 완료`
  - `FILE_PROGRESS|filename|95|결과 저장 완료`
  - `FILE_DONE|filename` 수신 시 100%
- 향후 Google Drive 업로드가 붙으면 업로드 단계도 현재 파일 진행률에 포함할 수 있다.
  - 로컬 저장 완료: 90%
  - Drive 업로드 시작: 92%
  - Drive 업로드 완료: 100%
- 다만 1차 진행률 개선에서는 Google Drive 업로드 전 단계이므로 로컬 전사/저장 기준까지만 반영한다.

### REMAINING TIME 예측 개선
- 현재 REMAINING TIME이 실제보다 과하게 길게 표시되는 문제가 있다.
- 예: 표시 약 25분, 실제 전사 시간 약 7분.
- 부동산학개론 실측값은 음원 길이 54분 59초, 실제 전사 소요시간 약 10분이다.
- 부동산학개론 실측 ratio는 `10분 / 54.98분 ≈ 0.18`이다.
- 현재 기본값이 실제 RTX 5060 + fp16 환경 기준으로 너무 보수적일 수 있으므로 초기 ETA 기본 ratio는 `0.18` 수준을 우선 검토한다.
- 전사 완료 후 실제 처리시간과 음원 길이를 비교해 실측 ratio를 계산한다.
- 공식: `실제 처리시간 / 음원 길이 = 실제 전사 ratio`
- 예: 음원 50분, 실제 전사 7분이면 `ratio = 7 / 50 = 0.14`
- 첫 파일은 기본 ratio로 예측하고, 이후 파일부터는 최근 완료 파일의 실측 ratio를 반영한다.
- 최근 3~5개 평균 또는 누적 평균을 사용하는 방향을 검토한다.
- CURRENT PROGRESS가 계속 0%에 머물면 elapsed 기반 ETA 보정도 부정확해지므로 CURRENT PROGRESS 개선과 REMAINING TIME 개선은 함께 고려한다.

## 6. Google Drive API 인증 설계
1. **방식**: OAuth Desktop App 기반 사용자 인증 (최초 1회 브라우저 로그인 필요)
2. **권장 Scope**: `https://www.googleapis.com/auth/drive.file` (앱이 직접 생성/사용한 파일 및 폴더만 접근 권한 부여). 보안상 전체 권한(`drive`)은 1차 구현에서 제외한다.
3. **토큰 관리**: 최초 로그인 시 획득한 token은 파일로 저장하여 이후 재사용하며, 만료 시 Refresh token을 이용해 자동 갱신 처리한다.

### Credentials / Token 보관 정책
보안 및 깃허브 업로드 방지를 위해 다음 원칙을 반드시 준수한다.
- `credentials.json` 및 `token.json`은 Git 커밋 및 업로드 절대 금지.
- **권장 보관 경로 (A안)**: 사용자가 GUI에서 `credentials.json` 파일을 선택하면 `%APPDATA%\전사도우미\` 내부에 안전하게 복사하여 사용. (예: `%APPDATA%\전사도우미\google_drive_token.json`)

## 7. 주요 구현 함수 설계

### 7.1. Drive 경로 생성 함수 (`build_drive_folder_path`)
- **역할:** 파일명을 기반으로 Google Drive 내의 업로드 대상 폴더 경로(리스트)를 반환한다.
- **반환 경로 예시:**
  `[ "2026 제37회 공인중개사 자격시험", "전사자료", 과정명, 과목폴더, 주차폴더 ]`
- 강의별 별도 하위 폴더는 생성하지 않는다.

### 7.2. 번들 업로드 함수 (`upload_transcription_bundle`)
- **설명:**
  - `upload_transcription_bundle(audio_path, include_mp3=True, service=None)`
  - MP3 파일명을 기준으로 같은 stem을 공유하는 `.txt`, `.json`, `.srt` 파일을 찾는다.
  - Drive에서는 과정/과목/주차 폴더까지만 생성 또는 탐색한다.
  - 해당 주차 폴더 안에 MP3/TXT/JSON/SRT 4종 파일을 직접 업로드한다.
  - 강의별 별도 폴더는 생성하지 않는다.

## 8. 중복 파일 처리 정책
동일한 주차 폴더 안에 동일한 이름의 파일이 이미 존재하는 경우의 처리 방법:

**추천 정책 (Update or Create):**
1. 같은 주차 폴더 안에 같은 파일명이 발견되면 **새 버전으로 업데이트 (Update)**.
2. 없으면 **새로 업로드 (Create)**.
3. (안전 대안) 1차 구현 시나 업데이트 실패에 대비하여 동일 파일명 발견 시 `_dup1`, `_dup2` 식의 Suffix를 붙여 **새 파일로 안전하게 생성 (Create)** 하는 대안 옵션도 유지.

## 9. 업로드 실패 처리 정책
- **전사 우선 원칙**: Google Drive 업로드 실패가 로컬 전사 저장 단계나 전체 작업의 실패로 이어지지 않아야 한다. 로컬 결과물은 무조건 안전하게 보존된다.
- **부분 실패**: MP3 업로드가 실패(네트워크 등)하더라도 TXT/JSON/SRT 업로드는 멈추지 않고 계속 진행한다.
- **재시도**: 네트워크 오류 시 N회(예: 3회) 재시도 로직을 적용한다.
- **인증 오류**: Refresh 실패 시 사용자에게 재로그인(GUI 알림/로그)을 안내한다.
- **로깅**: 파일별 업로드 성공/실패 여부를 철저히 로그로 남긴다.
